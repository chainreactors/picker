# **责任边界划分的核心逻辑**

* **谁写的代码，谁负责安全**：这是软件工程的基本原则。
* **支持角色提供保障，并非最终责任人**：Security Champions 和安全团队支持研发团队提高安全能力，而非替代其承担责任。

# 积极的安全文化建设

* **安全是业务的重要组成部分：**作为业务的推动力，高管提供必要的支持。
* **安全是每个人的工作：**研发团队负责其产品的安全性，确保安全解决方案嵌入到设计与实现中，安全团队提供工具、指导和发布前的安全验证支持。
* **明确的流程、工具和规则是可持续性安全必要手段：**可控的输入生成持续的输出，实现安全问题的系统性解决和持续改进。

# 通过 Security Champions 拓宽应用安全能力边界

## 什么是 Security Champions？

* Security Champions 是拓宽应用安全能力的一种机制，目标是将安全能力延伸到研发团队。
* 核心是通过培训研发工程师，让他们在团队中承担 Security Champion 的角色。
* 通过对研发团队中的开发者进行培训，提升其在早期阶段以安全为核心的设计思维。
* 注：Security Champion 是在研发团队的安全接口人，接受来自安全团队的关于安全工具的使用和流程方面的培训，承担团队内部的威胁建模、安全评审和安全工具 on board、漏洞修复指导等工作，业界通常将其称为 Security Champions、Security Advocates、Security Innovators 和 Security Drivers 等，在这里我们统一将其称为 Security Champions。

## 为什么需要 Security Champions？

* 安全团队资源有限，通常是实施安全评审的人员不足，难以将安全能力面面俱到的覆盖到所有开发工作。
* 产品研发周期中的安全瓶颈会延迟功能发布，拉长新功能交付的时间，影响业务目标。
* 提前解决安全问题比事后修复更高效、更经济。

## 试点团队

1. **先选择与安全团队配合良好的兄弟部门：**这类研发团队往往有较好的安全意识和实践经验，已经应用了部分安全团队提供的解决方案（如 SAST、DAST、SCA 等），适合作为早期试点团队，快速验证和展示成果。
2. **再选择高影响力团队量化提升的效率：**受公司范围内安全流程影响较大，通过该项目，可以显著提升敏捷性和交付效率。
3. **最后再选择高风险团队，作为标杆客户：**负责核心业务和敏感数据的团队，需要更高标准的安全评审，有效提高公司的安全水位。

## 衡量项目的成果

* 新功能发布的平均时间（是不是比之前更快了？快了多少？）
* 平均安全漏洞数（漏洞是不是更少了？）
* 研发工程师花费在安全问题上的平均时间（是不是节约了更多的开发资源？）
* Security Champion 参加安全工具、流程培训的比例（衡量 Security Champion 的参与度）
